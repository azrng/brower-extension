# Boss 直聘职位技能分析插件界面设计要求

## 1. 文档目标

本文档只描述插件的界面设计要求，不描述采集逻辑、AI 实现细节和代码结构。

该插件界面采用 **页面内右侧悬浮 sidebar** 方案，在 Boss 直聘搜索页中提供任务控制、配置管理、进度展示、结果查看和 JSON 导出能力。

## 2. 设计原则

- 界面必须嵌入在 Boss 页面右侧，不跳出独立业务页面
- 界面以任务操作为主，尽量减少复杂层级
- 用户应能在一个侧栏内完成配置、启动、查看结果、导出
- 界面应明确区分 `未开始`、`运行中`、`已完成`、`失败/已停止` 四种状态
- 界面应优先保证信息清晰，而不是追求复杂视觉效果
- 不明确的交互细节必须先确认，不能自行假设

## 3. 适用页面

界面仅需支持以下页面场景：

- Boss 直聘搜索结果页

当前阶段不要求在其他页面保持完整功能，但如果用户从搜索页启动任务后打开职位详情页，插件内部流程仍可继续执行。

## 4. 总体界面结构

右侧悬浮 sidebar 建议由以下区域组成，自上而下排列：

1. 顶部标题区
2. 任务控制区
3. 参数配置区
4. 任务进度区
5. 结果展示区
6. 导出区
7. 状态提示区

侧栏应固定在页面右侧，避免遮挡核心职位列表内容。支持展开/收起，但默认展开。

## 5. 顶部标题区

### 5.1 目标

用于明确插件用途和当前状态。

### 5.2 必要元素

- 插件名称，例如：`职位技能分析`
- 当前状态标签
- 收起/展开按钮
- 关闭按钮（仅关闭侧栏 UI，不影响插件安装）

### 5.3 状态标签

状态标签至少包含以下值：

- `未开始`
- `运行中`
- `已完成`
- `已停止`
- `失败`

状态颜色应明显区分，但不要求复杂视觉系统。

## 6. 任务控制区

### 6.1 目标

让用户清楚地启动、停止当前任务。

### 6.2 必要按钮

- `开始识别`
- `停止任务`

### 6.3 交互要求

- 未开始时，`开始识别` 可点击，`停止任务` 默认禁用
- 运行中时，`开始识别` 禁用，`停止任务` 可点击
- 停止后可再次开始新任务
- 已完成后允许重新开始任务
- 如果参数缺失，点击开始时应给出明确提示，不能静默失败

## 7. 参数配置区

### 7.1 目标

让用户在侧栏中直接配置任务执行参数和 AI 接口参数。

### 7.2 配置项

必须支持以下字段：

- `API Base URL`
- `API Key`
- `Model`
- `前几页 n`
- `请求超时`
- `处理间隔`

### 7.3 字段要求

#### API Base URL

- 文本输入框
- 必填
- 允许用户填写兼容 OpenAI 规范的接口地址

#### API Key

- 密钥输入框
- 必填
- 默认以掩码显示
- 需要支持显示/隐藏切换

#### Model

- 文本输入框或可编辑下拉框
- 必填
- 不绑定具体平台模型名

#### 前几页 n

- 数字输入框
- 必填
- 最小值建议为 `1`
- 含义必须明确标注为“最多处理前几页搜索结果”

#### 请求超时

- 数字输入框
- 单位明确，例如秒或毫秒
- 必填

#### 处理间隔

- 数字输入框
- 用于控制职位之间的处理节奏
- 必填

### 7.4 配置交互要求

- 支持保存配置
- 下次打开插件时应自动回显上次保存的配置
- 开始任务前应校验必填项
- 校验失败时，应在字段附近或状态区显示明确错误信息

## 8. 任务进度区

### 8.1 目标

让用户实时知道当前任务执行到哪里。

### 8.2 必要展示内容

- 当前总状态
- 当前处理页数 / 总页数
- 当前处理职位数 / 已发现职位数
- 当前阶段说明
- 最近一条处理记录

### 8.3 当前阶段说明建议值

至少支持以下阶段文案：

- `正在读取当前页职位列表`
- `正在打开职位详情`
- `正在提取职位内容`
- `正在调用 AI 分析`
- `正在汇总技能结果`
- `任务完成`

### 8.4 进度表现形式

建议同时提供：

- 简单进度文本
- 线性进度条

进度区需要在长任务中持续更新，不能只在完成后一次性刷新。

## 9. 结果展示区

### 9.1 目标

让用户直接在侧栏中查看 AI 汇总结果，不必先导出文件。

### 9.2 必要模块

结果区至少包含以下内容：

- 技能关键词列表
- 分类汇总
- 技能频次统计
- 同义词归一结果
- 总体总结

### 9.3 展示要求

#### 技能关键词列表

- 以标签或列表形式展示
- 每个关键词应清晰可读
- 数量较多时允许折叠或滚动

#### 分类汇总

- 按 AI 返回的分类分组展示
- 每组下显示对应技能项
- 分类名称允许动态变化，不使用固定枚举限制

#### 技能频次统计

- 展示技能名称与出现次数
- 建议按频次倒序展示
- 频次高的技能应更容易被识别

#### 同义词归一结果

- 展示“原词 -> 标准词”的映射关系
- 例如：`JS -> JavaScript`
- 应让用户看出哪些词被合并过

#### 总体总结

- 使用 1 到 3 段简短文字总结岗位技能画像
- 用于帮助用户快速理解整体招聘倾向

## 10. 导出区

### 10.1 目标

让用户把最终汇总结果导出为 JSON 文件。

### 10.2 必要元素

- `导出 JSON` 按钮

### 10.3 交互要求

- 未产出有效结果前，导出按钮默认禁用
- 有结果后允许导出
- 导出的内容只包含汇总结果，不导出原始职位全文
- 导出成功后应给出提示
- 导出失败时应明确提示失败原因

## 11. 状态提示区

### 11.1 目标

统一展示提示、错误、警告和成功信息。

### 11.2 提示类型

至少支持以下类型：

- 成功提示
- 错误提示
- 警告提示
- 普通提示

### 11.3 典型提示文案

- `请先填写 API Base URL`
- `请先填写 API Key`
- `当前页面未识别到职位列表`
- `未找到下一页，任务已提前结束`
- `AI 请求超时`
- `任务已停止`
- `JSON 导出成功`

## 12. 界面状态切换要求

### 12.1 未开始

- 展示配置表单
- 允许用户编辑参数
- 无结果内容或展示空状态

### 12.2 运行中

- 锁定关键配置项，避免任务中途被随意修改
- 突出显示当前进度和当前处理阶段
- 允许点击停止任务

### 12.3 已完成

- 展示完整汇总结果
- 允许导出 JSON
- 允许重新开始新任务

### 12.4 已停止

- 保留已采集到的阶段性结果
- 标明任务是被手动停止还是异常中断
- 允许重新开始任务

### 12.5 失败

- 明确显示失败原因
- 不隐藏已经获得的中间结果
- 提供可恢复操作，例如重新开始

## 13. 空状态要求

当没有可展示内容时，需要有明确空状态，而不是空白区域。

至少包括以下空状态：

- 未开始任务
- 当前页未识别到职位数据
- 任务尚未生成分析结果
- 已停止但暂无有效结果

## 14. 可用性要求

- 侧栏宽度应适中，既能容纳结果信息，又不要大面积遮挡页面
- 长内容区域应支持内部滚动
- 关键按钮位置要固定且容易找到
- 输入框标签必须清晰，不使用模糊命名
- 文案尽量直接，不使用含糊描述

## 15. 非目标范围

当前界面阶段不要求支持以下能力：

- Excel 导出
- 多站点切换
- 多任务并发管理
- 历史任务列表
- 用户登录系统
- 复杂图表分析
- 手动编辑 AI 结果

## 16. 建议补充的界面稿输出物

后续如果进入设计或开发阶段，建议继续补充以下内容：

- sidebar 线框图
- 关键状态切换图
- 配置表单字段校验规则表
- 结果区 JSON 对应展示映射表
- 组件拆分清单

## 17. 结论

当前 README 中的内容适合描述总体需求、业务范围和执行规则，但对于界面设计来说还不够细。

因此需要单独维护本界面设计文档，用于明确：

- 侧栏结构
- 按钮行为
- 配置表单
- 进度反馈
- 结果展示
- 导出交互
- 状态切换

后续如果要进入实现阶段，应优先以本文档作为界面开发依据。
